# 为 TvBox_WZ 项目添加完整日志系统

## 摘要

项目当前完全没有使用 `logging` 模块，所有诊断信息都通过零散的 `print()` 输出，格式不统一（`[错误]`、`[信息]`、`[JsonSpider]`、`[ParsesLoader]` 等前缀混用），且不持久化——程序关闭后信息丢失，无法事后排查问题。

本次任务：引入 Python 标准库 `logging`，在所有关键数据流转点添加日志，同时写入控制台和滚动日志文件（`logs/tvbox.log`），让用户遇到问题时能带着日志来排查。**不修复任何现有 bug**，只加日志。

## 当前状态分析

### 日志现状
- 全项目 **0 处** 使用 `logging` 模块
- 24 个 Python 文件中共 694 处 `print()` 调用
- 诊断性 print 和用户界面 print 混杂，难以区分
- 异常路径只有部分被打印，多数 `except` 块静默 `pass` 或只返回错误字符串
- `parses_loader.py` 和 `mock_spider.py` 完全没有异常保护
- `main.py` 的 `try/finally` 缺少 `except` 兜底，致命异常无法被记录

## 设计决策

### 日志基础设施
- **使用标准库 `logging`**：零新增依赖
- **新建 `utils/logger.py`**：集中配置日志系统，提供 `setup_logging()` 和 `get_logger()` 两个函数
- **双输出**：控制台（INFO 级，简洁格式）+ 文件（DEBUG 级，详细格式含文件名行号）
- **滚动文件**：`logs/tvbox.log`，单文件 5MB，保留 3 个备份，UTF-8 编码
- **日志目录**：项目根目录下 `logs/`，自动创建，加入 `.gitignore`

### 日志格式
- 控制台：`2026-07-09 14:30:15 [INFO] config_loader: 成功解析 48 个视频源`
- 文件：`2026-07-09 14:30:15,123 [INFO] config_loader (config_loader.py:181): 成功解析 48 个视频源`

### print vs logger 的划分原则
- **保留为 `print()`**：用户界面输出（菜单、分隔线、提示语、播放状态反馈等 CLI 交互内容）
- **改为 `logger`**：诊断信息（模块内部状态、网络请求细节、数据解析过程、错误堆栈等）
- **错误双重输出**：关键错误同时 `print`（用户即时反馈）+ `logger.error`（持久化含堆栈）

### 日志级别使用规范
| 级别 | 使用场景 |
|------|----------|
| DEBUG | HTTP 请求 URL/参数/响应状态码/耗时、数据解析中间步骤、解析器选择过程 |
| INFO | 配置加载完成、爬虫创建、解析器注册、播放器检测、播放启动 |
| WARNING | 字段类型转换失败回退、未匹配到解析器、空数据兜底、不支持的数据类型 |
| ERROR | 网络请求失败、JSON 解析失败、文件不存在、播放器启动失败 |
| CRITICAL | main.py 兜底的未捕获致命异常 |

## 具体改动方案

### 1. 新建 `utils/logger.py`（日志配置模块）

新文件，提供：
- `setup_logging(log_dir="logs", console_level=INFO, file_level=DEBUG)`：初始化日志系统，配置控制台和滚动文件 handler
- `get_logger(name)`：获取命名 logger（封装 `logging.getLogger`，方便统一管理）
- 首次调用 `setup_logging` 时自动创建 `logs/` 目录
- 使用 `RotatingFileHandler`（maxBytes=5MB, backupCount=3, encoding=utf-8）

### 2. 修改 `main.py`（入口 + 全局异常兜底）

- 在 `main()` 最开头调用 `setup_logging()`
- 启动时 `logger.info("TvBox 启动，config_path=%s", config_path)`
- 将 `try/finally` 改为 `try/except Exception/finally`：
  - `except` 中 `logger.critical("程序异常退出", exc_info=True)` 并 re-raise
  - `finally` 中 `logger.info("服务器已关闭")` + `stop_server()`
- 记录 `start_server()` 和 `CliApp` 创建/运行过程

### 3. 修改 `utils/config_loader.py`（配置加载）

- 文件顶部 `logger = logging.getLogger(__name__)`
- 将所有 18 处诊断 print（`[错误]`/`[信息]`/`[警告]`）改为对应级别的 `logger` 调用
- `load_from_file` 入口：`logger.info("从文件加载配置: %s", file_path)`
- `load_from_url` 入口：`logger.info("从 URL 下载配置: %s (超时 %ss)", url, timeout)`
- `_parse_sites` 循环中：`logger.debug("解析到视频源: key=%s, name=%s, type=%s", ...)`
- `_parse_parses` 循环中：`logger.debug("解析到解析源: name=%s, type=%s, url=%s", ...)`
- 所有 `except` 块：`logger.error("...", exc_info=True)` 记录完整堆栈

### 4. 修改 `utils/parses_loader.py`（解析器加载）

- 添加 `logger = logging.getLogger(__name__)`
- 将 3 处 print 改为 logger 调用
- `create_parser`：DEBUG 记录 type 和创建结果；WARNING 记录不支持的类型
- `load_into_manager`：DEBUG 记录每个注册过程；INFO 记录最终汇总
- **新增异常保护**：`create_parser` 中加 try/except，构造失败时 `logger.error("解析器创建失败: %s", name, exc_info=True)` 返回 None
- `load_from_config`：INFO 记录来源和 parses 数量

### 5. 修改 `spiders/json_spider.py`（JSON 爬虫）

- 添加 `logger = logging.getLogger(__name__)`
- `_get` 方法（最关键）：

## 假设与决策

1. **不修复任何现有 bug**：用户明确要求只加日志，不修 bug。`vod_info.py:194-196` 的 `@` 分隔符 bug 保持原样，但会加 WARNING 日志暴露它。
2. **不新增第三方依赖**：仅用 Python 标准库 `logging` + `logging.handlers.RotatingFileHandler`。
3. **保留所有现有 UI print**：CLI 界面的 print 输出不变，避免影响用户体验。
4. **诊断 print 转为 logger**：`[错误]`/`[信息]`/`[JsonSpider]` 等诊断性 print 转为 logger 调用，既显示在控制台又写入文件。
5. **日志文件位置**：`logs/tvbox.log`（项目根目录下），与 `main.py` 同级。
6. **日志编码**：UTF-8，确保中文不乱码。
7. **测试文件不加日志**：`Tests/` 目录下的测试脚本保持原样，避免干扰测试输出。
8. **demo 脚本不加日志**：`demo_play.py`、`demo_parses.py` 保持原样。
9. **server/mock_api_server.py 暂不加日志**：模拟服务器是测试辅助工具，优先级低，本次不改动。
